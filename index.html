First, five words

The vocabulary, in one screen

If you have a STEM degree but have never touched AI tools, these five terms unlock everything below. Read them once.

Model
The raw AI brain itself, like GPT-5 or Claude. A giant math function trained on text. It is frozen: you cannot change how it thinks, only what you feed it.
Agent
A model put to work in a loop: read the task, run a command, look at the result, decide the next step, repeat. An AI "coding assistant" like Claude Code or Cursor is an agent.
Context window
The agent's short-term memory: the limited amount of text it can hold at once. Big, but finite. Fill it up and older details fall out the back.
Token
A chunk of text, roughly ¾ of a word. The unit everything is measured and billed in. The context window is measured in tokens.
Harness
Everything around the model that isn't the model: the instructions, the files, the tools, the memory, the tests. The whole work environment. This entire course is about building a good one.
The core reframe

The model is rarely the bottleneck

Your agent botches a task. The obvious fix is a smarter, pricier model. Usually, that is the wrong move. Even the best models pass only 50–60% of real software tasks on SWE-bench (a public test set of actual GitHub bug reports), and the gap between "smart on a test" and "reliable on your repo" is almost never about raw intelligence. It is about the environment you put the intelligence in.

bare
🐎

A thoroughbred with no saddle

Drop a champion racehorse in a field with no saddle, no bridle, no reins. World-class engine, zero control. It bolts, stumbles, goes nowhere useful. The horse is not the problem.

harnessed
🏇

The same horse, properly tacked

Add a good saddle, reins, and clear signals and the same animal wins races. The equipment ("tack") didn't make it faster. It made the speed usable.

The model is the horse. The harness is the tack. This course teaches you to build the tack, so the intelligence you already have actually crosses the finish line.

It is measurable

A separate study had 3 engineers write zero code by hand for 5 months and let the agent build ~1,000,000 lines across 1,500 pull requests. Early progress was slow not because the model was weak, but because the scaffolding wasn't there yet.

It is repeatable

A team adding endpoints to a real app saw the bare agent burn 40% of its memory just figuring out the codebase, then ship code with runtime errors. With a harness, the same model passed every run.

It is cheap to fix

The punchline from the course: one good instructions file can beat a model upgrade, and it costs nothing. Most teams walk right past it.

Why the repo is everything

An agent has only three senses

Here is the most important mental model in the course. An agent cannot ask a colleague, scroll Slack, or read your mind. It can perceive exactly three things. Everything else, no matter how obvious to you, simply does not exist for it.

👁️ What it CAN see

📝
The task you typed and its system prompt
📂
Files in the repo it can open and read
⚡
Tool output, like what a command printed

🚫 What it CANNOT see

💬
That Slack thread where you decided the architecture
📊
The Confluence page nobody updated since 2023
🎫
The Jira ticket with the real requirements
🧠
The rule that lives only in a senior engineer's head
If it isn't in the repo, it doesn't exist.

So the fix is blunt: make the repository the single source of truth. Write the decisions down, put them next to the code, and a fresh agent can answer "what is this, how do I run it, how do I check it" from files alone.

The whole course

The failures between a slick demo and something you can ship

Getting an agent to a slick demo is easy. Getting to something you can actually ship is where it falls apart: it forgets yesterday's decisions, declares victory too early, and leaves the seams broken. Each failure below comes with why it happens and the fix. Four acts. Tap any to open it.

1

Foundations

why agents fail & what to build
01Why capable agents still fail›
The trapA smart model fails, so you blame the model and pay for a bigger one.
The fixCheck the harness first. Failures cluster in 5 fixable layers, not in the model's brain.
Like you're 20A straight-A student fails the exam because nobody told them the room, the time, or the syllabus. Not a brains problem.
02What a harness actually is›
The trapThinking "harness" = one clever prompt file.
The fixIt is 5 subsystems: instructions, tools, environment, state, feedback. Feedback gives the most bang per buck.
Like you're 20A new hire needs more than a welcome note. They need login access, the wiki, a desk, and a way to check their work.
03The repo is the only source of truth›
The trapCritical knowledge lives in Slack, Jira, and people's heads, all invisible to the agent.
The fixWrite decisions into files in the repo, next to the code they govern. Pass the "fresh session test."
Like you're 20If "this road is closed" only lives in one guy's head, everyone keeps crashing into it. Put up a sign.
2

Memory & setup

surviving long, multi-day work
04Why one giant instruction file fails›
The trapThe rules file grows to 600 lines. Only a third matters, and the key rule is buried in the middle where the model skims past it.
The fixKeep a short entry file (a router, ~100 lines). Push details into topic docs, revealed only when needed.
Like you're 20A good menu is one page with sections, not a 50-page novel. You read the section you need.
05Why long tasks lose the plot›
The trapMemory fills up, the session resets, and the new session forgets why earlier choices were made. It redoes or contradicts past work.
The fixWrite a progress log and a decisions log before "clocking out." The next session reads them and resumes in 3 minutes, not 15.
Like you're 20A nurse handing off a shift writes notes. Without them, the next nurse re-runs every test from scratch.
06Why setup needs its own phase›
The trapMixing "set up the project" with "build features" means the agent rushes the boring setup and pays for it later.
The fixFirst session builds only the foundation: runs, one test passes, progress file and task list exist. Start from a template, not an empty folder.
Like you're 20Cooks do "mise en place," chopping and laying out everything, before the heat is on. Prep is its own job.
3

Focus & finishing

do less, actually complete it
07Why agents bite off too much›
The trap"While I'm here, let me also fix this..." Six features started, zero finished. Adding a task is free, so it never stops.
The fixWIP = 1. One task at a time. It cannot start the next until the current one passes a real test. Less code, more done.
Like you're 20Juggle six balls at once and you drop all six. Finish one, then pick up the next.
08Why "done" needs a definition›
The trap"Done" means different things to you and the agent. Loose notes like "cart mostly done" are useless to the next session.
The fixA machine-readable feature list: each item has a behavior, a verify command, and a state. The agent can't flip the state itself, the passing test does.
Like you're 20A contractor's checklist with inspections, not "house mostly built, trust me."
09Why agents declare victory too early›
The trapModels are systematically overconfident. "It's done!" really means "the code looks fine to me," which is not the same as "it works."
The fixTake the verdict out of the agent's hands. A separate, nitpicky checker runs real proof. Worker and inspector must be different roles.
Like you're 20You don't grade your own exam. An independent examiner does.
4

Proof & hygiene

trust, visibility, and clean handoffs
10Why end-to-end tests change results›
The trapEach part passes its own isolated test, but the seams between them break. Isolation is exactly what hides the bug.
The fixRun the whole pipeline for real. Knowing it will be tested end-to-end even changes how the agent writes the code in the first place.
Like you're 20Every part of the car passes on the bench, but nobody drove it. Take it around the block.
11Why you must see inside the run›
The trapNo logs, no scorecards. So "good or bad" becomes a vibe, and retries are blind guessing.
The fixBuild in observability: runtime signals (what happened) plus decision artifacts like a scoring rubric (why it was accepted). One study cut a task from 45 to 15 minutes purely from this.
Like you're 20A doctor with no thermometer or chart is just guessing. Instruments turn opinion into evidence.
12Why every session must end clean›
The trapMess accumulates. Broken build, stray debug files, no notes. "Clean it up later" means never. Entropy is the default.
The fix"Clean state" is part of done: build passes, tests pass, progress logged, no junk, still launchable. Periodically delete harness rules the model has outgrown.
Like you're 20One coffee cup left out becomes ten by Friday. A clean kitchen stays clean only if each cook cleans up.
Worth playing with

The habits that quietly kill a coding session

Overreaching and faking "done" are the most common failures, and the most fun to see. Try both.

1. Overreach vs. WIP = 1

Asked to "add login," a greedy agent spirals into six half-built features. Flip on the one-task-at-a-time rule (WIP = 1).

2. The completion gate

The agent claims it's finished. Don't take its word. Make the claim pass three real gates.
①
Syntax & static checksdoes it even compile?
②
Runtime behaviordoes it actually run?
③
End-to-end systemdoes the whole flow work?

What a harness looks like on disk

AGENTS.mdthe map & the rules
# AGENTS.md: the agent reads this FIRST, every session

## Startup workflow (run before writing any code)
1. pwd                          # where am I?
2. read claude-progress.md       # what happened last time?
3. read feature_list.json, pick the top unfinished feature
4. git log --oneline -5         # recent history
5. ./init.sh                    # install + verify baseline is green

## Working rules (the guardrails)
- One feature at a time. Finish it before starting the next.
- Code written is NOT the same as feature done.
- Do not silently change the verification rules to pass.
- Prefer durable files over chat summaries.

## Definition of Done (all must hold)
- behavior implemented AND its verify command actually ran
- evidence recorded in feature_list.json
- repo still starts cleanly from ./init.sh

## Topic docs (read on demand, keep THIS file short)
- docs/api-patterns.md      # when adding endpoints
- docs/database-rules.md    # when touching the DB
// feature_list.json: the single source of truth for "what's done"
{
  "rules": {
    "single_active_feature": true,
    "passing_requires_evidence": true
  },
  "features": [
    {
      "id": "chat-001",
      "title": "Create a new conversation",
      "user_visible_behavior": "Click New Chat, see an empty thread.",
      "status": "not_started",   // not_started | in_progress | blocked | passing
      "verification": [
        "Open the app.",
        "Click New Chat.",
        "Verify a new conversation appears in the sidebar."
      ],
      "evidence": []        // only filled once it actually passes
    }
  ]
}
# claude-progress.md: the shift-handoff note

## Current Verified State
- Repository root: /app
- Standard startup: ./init.sh
- Standard verification: npm test
- Highest-priority unfinished feature: chat-002 (send a message)
- Current blocker: none

## Session Log
### Session 003
- Date: 2026-06-17
- Goal: implement chat-001 (new conversation)
- Completed: chat-001, verified, marked passing
- Evidence: commit a1b2c3d, manual steps all green
- Known risk: sidebar flickers on first load, not blocking
- Next best step: start chat-002, do NOT refactor the sidebar yet
#!/usr/bin/env bash
# init.sh: one command to get to a known-good baseline
set -euo pipefail

INSTALL_CMD=(npm install)
VERIFY_CMD=(npm test)
START_CMD=(npm run dev)

# 1. install deps
"${INSTALL_CMD[@]}"

# 2. prove the baseline is GREEN before any new work
"${VERIFY_CMD[@]}"

# 3. only launch the app if explicitly asked
if [ "${RUN_START_COMMAND:-0}" = "1" ]; then
  "${START_CMD[@]}"
fi

Notice the pattern: every file pushes the agent to read state first, do one thing, prove it, write it down, leave it clean. That loop is the whole game.

Keep this

The whole course on one card

Remember these and you have the spine of the whole discipline. The rest is detail.

01

Blame the harness first

When the agent fails, fix the environment before reaching for a bigger model. The bottleneck is almost never the brain.

02

Five subsystems

Instructions, tools, environment, state, feedback. Feedback (real verify commands) is the cheapest, highest-return one.

03

The repo is reality

If it isn't in a file the agent can open, it doesn't exist. Write decisions down, next to the code.

04

A map, not a manual

Short entry file that routes. Details live in topic docs, revealed on demand. Buried rules get ignored.

05

Memory is a file

The window is finite. Long work survives only if each session logs progress and decisions for the next one.

06

WIP = 1

One task at a time, finished and verified, before the next. Less code, more shipped. Quality beats volume.

07

"Done" is a passing command

Not a feeling. Every feature carries a behavior, a verify command, and a state the agent can't fake.

08

Separate worker and checker

Models grade themselves too kindly. An independent, nitpicky verifier and real end-to-end runs catch the lies.

09

Leave it clean

Clean state is part of done. Entropy is the default; only deliberate cleanup each session fights it.

The takeaway

Reliability was never a smarter model. It's a better harness around the one you already have.

The intelligence is already in the box. Harness engineering is the unglamorous, compounding work of building an environment where that intelligence can't wander off, can't lie about being done, and can't forget what it learned yesterday.
